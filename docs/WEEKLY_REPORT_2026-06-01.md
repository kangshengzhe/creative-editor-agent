## 一、本周总览

本周项目从"能生成文案的 v1"推进到"**多市场可投放、合规、可运营**"的阶段。核心做了三件大事：

1. **落地业务硬指标**：每条广告组精确产出 15 标题 / 10 描述，三层差异化、母语生成、关键词本地化全部打通。
2. **打磨真实可用性**：性能优化、运营审核翻译、中断按钮、自定义约束等，让香港运营团队真正能上手。
3. **系统性攻克多语言质量问题**：截断、关键词匹配两大类问题，按 20 种语言各自的语言学特性逐一解决，消除了大量"误报"。

---

## 二、按主题的具体进展

### 1. 业务硬指标：15 标题 / 10 描述 + 三层差异化 ✅
- 配额表驱动生成（标题15、描述10、CTA5、长文案5），超量生成后排序截断到**精确数量**。
- **三层差异化**：语义相似度去重（sentence-transformers，阈值校准 0.60）+ 创意角度拆分 + 排序去重，保证 15 条标题彼此不雷同。
- **母语生成**：20 种语言直接用母语创作（非翻译英文稿），按目标市场切换。
- 数量不足时，按业务方确认的"数量优先"策略从近义储备池回填，保证精确 15/10。

### 2. 关键词本地化 ✅
- 通用词翻译、品牌词保留（如 topup → recarga / пополнение），全部非英语市场生效。

### 3. 运营可用性增强 ✅
- **审核翻译**：交付文案翻成简体中文/繁体中文/英语，供香港运营审核（运营母语是中文）。
- **自定义约束**：must_include / must_avoid（提示词+硬过滤双保险）/ extra_instructions / 换一批，让运营像调 AI 一样精细控制。
- **中断按钮**：运营可随时停下修改 Brief。
- **提示中文化**：所有警告提示从英文码改为中文，运营看得懂。

### 4. 性能优化 ✅
- LLM 客户端复用连接池（不再每次新建握手）。
- 标题生成的补位阶段从"串行"改为"并发波次"。
- 批量生成（标题+描述+CTA）实测约 **72 秒**（此前一度 145 秒）。

### 5. 多语言质量：本周重点攻坚 ⭐

按"诚实排查、逐语言处理"的原则，系统性解决了两类影响 20 种语言的问题：

**(A) 文案截断不再切半句**
- 问题：机械按字符硬切，切出 `value, s`、`Topup Bo` 这种残缺文案。
- 方案（"宁短勿截"）：① LLM 生成时控长写完整句；② 优先保留限制内的完整文案，超限丢弃；③ 兜底才按词边界截断（永不切半词）。无空格语言（中日泰高棉）按显示宽度处理。

**(B) 关键词匹配按语言特性区分**
- 问题：严格整词匹配对很多语言**误报"关键词缺失"**——阿语冠词粘连（الشحن）、中日泰无空格、俄语变格（бонус→бонусом）、西语复数（recarga→recargas）、土耳其后缀、印尼/菲律宾词缀。
- 方案：按**书写系统 + 目标语言**选匹配策略（子串 / 西里尔词干 / 拉丁词干 / 词内子串），英语与越南语保持严格（play ≠ player）。覆盖全部 20 种语言。

### 6. 文档沉淀 ✅
- 运行指南、工作流思维导图、广告平台政策、市场语言对照表等，便于交接与例会查阅。

## 三、本周最大的难点与攻坚过程 ⭐

> 这部分是本周投入最多、最能体现工作含量的地方。难点不在"写功能"，而在**反复发现"看起来好了、其实没好"的隐蔽问题**，并逐层挖到真正的根因。

### 难点 1：多语言关键词匹配 —— 一个问题挖出 20 种语言的坑（最难）

- **表面现象**：阿拉伯语市场报"关键词 topup 没放进文案"，但审核翻译里明明有"充值"的意思，自相矛盾。
- **挖掘过程**：写诊断脚本用真实数据验证，发现是阿语定冠词 `ال` 与词粘连（`شحن`→`الشحن`），严格整词匹配认不出。修完阿语后**主动反问"其他语言会不会也有"**，逐一排查 20 种语言，结果发现：中/日/泰/高棉（无空格）几乎全失效、俄语变格（бонус→бонусом）、西语复数、土耳其后缀、印尼/菲律宾词缀……是一个**普遍的语言学问题**。
- **难在哪**：① 每种语言的形态规则都不同，不能一刀切；② 拉丁字母语言无法靠"字符"区分（西语要宽松、英语要严格，都是拉丁字母），最后做了**架构改动**把"目标语言"传进匹配逻辑；③ 要在"消除误报"和"不引入新的误匹配"之间反复权衡。
- **结果**：按书写系统 + 语言形态分成 5 类匹配策略，覆盖全部语言，俄语误报从 6 条降到 0 条变格误判。

### 难点 2：文案被"生硬截断" —— 三次才找到对的思路

- **挖掘过程**：先以为是"切到单词中间"，改成按词边界截断；用户进一步指出"截断完还得是完整句子"。最终认识到**正确思路是从源头控制、超限就丢弃**，而不是事后补救截断。
- **难在哪**：这是一个"思路逐步纠正"的过程——每一版看似解决了，实则只解决了表象。最终方案（fit-first：优先完整文案、超限丢弃、兜底才截断）才真正符合"宁短勿截"的业务直觉。

### 难点 3：审核翻译"张冠李戴" —— 隐蔽的批量错位 bug

- **表面现象**：某条文案的中文翻译，内容对不上它的原文。
- **挖掘过程**：实测发现批量翻译时，LLM 会**把每条的编号标错**，导致翻译串到了别的候选上；还会漏掉列表最后一条。
- **难在哪**：这种 bug 不报错、不崩溃，只是"结果悄悄错位"，极难发现。最终改成**每条独立翻译**，从根上杜绝错位。

### 贯穿全周的方法论难点

- **"机制做了" ≠ "真的好用"**：本周大量时间花在**用真实 LLM 跑诊断脚本验证根因**，而不是凭代码逻辑想当然。很多问题都是肉眼看页面才发现的，必须沉下去用数据复现。
- **性能波动定位**：生成耗时在 77~187 秒间剧烈波动，排查后确认大头是 LLM 接口响应本身，程序能优化的是"串行等待"部分（已做并发化），但无法消除接口本身的慢——这个边界要诚实区分。

## 五、诚实标注的未验证项 / 待确认事项

> 以下是机制上已实现、但仍需进一步验证或业务方确认的，例会上可一并对齐：

1. **多语言翻译质量**：母语生成 + 审核翻译需对应语言的母语者抽查，无法保证每条都地道。
2. **真实平台过审效果**："常换常新"能否真正绕过平台误判，需在真实投放中验证。
3. **关键词匹配的取舍**：屈折语言用词干匹配会"略宽松"（如俄语 `счёт` 词干也匹配 `счётчик`），这是为消除误报接受的取舍；词干截取目前是启发式规则。
4. **广告政策细节**：博彩/充值类业务在各国的资质要求、CTA 字段精确上限等，需业务方/平台官方核实（已在 AD_POLICIES.md 列出待办）。
5. **剩余"真缺失"**：部分语言仍有少量文案未含全部关键词，属 LLM 生成质量问题（非匹配 bug），可后续强化提示词。
